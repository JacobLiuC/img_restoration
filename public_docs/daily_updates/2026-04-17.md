## 今日判断

- 图像编辑继续从单次提示转向多轮会话式工作流。
- 真实性约束正在从结果打分转向过程控制和局部修复。
- 通用编辑模型已经可以覆盖一部分 restoration，但前提是有明确 fidelity constraints 和迭代 refinement。
- 现在最值得投入的不是“更大模型”，而是理解、生成、验证、局部修补之间的链路工程。

## 重点论文详解

### 1. [CoVFT](https://arxiv.org/abs/2603.21077)

- 日期：2026-03-22
- 这篇在做什么：这篇论文系统讨论多模态大模型里视觉编码器到底该不该调，并把问题归因为视觉偏好冲突。
- 关键方法：作者引入 Context Vector Extraction 和 Contextual Mixture-of-Experts，把多模态上下文显式注入视觉侧微调过程。
- 为什么重要：论文给出的核心信号很强，经过 CoVFT 的 7B 模型平均表现超过 13B 对照，说明视觉侧优化还远没有吃满。
- 对系统设计的启发：如果你们后面做编辑理解与恢复一体化，不能只盯语言侧 prompt 和 planner，视觉侧的上下文感知适配同样可能决定上限。

### 2. [FlowInOne](https://arxiv.org/abs/2604.06757)

- 日期：2026-04-08
- 这篇在做什么：它想把文本描述、布局约束、编辑指令全部变成视觉提示，统一到 image-in image-out 的 flow matching 框架里。
- 关键方法：核心是把所有输入都视觉化，再用单一流模型完成生成、布局控制和视觉指令编辑，同时配套 VisPrompt-5M 和 VP-Bench。
- 为什么重要：这条线代表一个很新的方向：不再执着于跨模态拼接，而是把理解和生成尽量压到统一视觉空间里。
- 对系统设计的启发：如果未来想做 Gemini/Nano Banana 那种理解和编辑合一系统，FlowInOne 提醒我们可以考虑视觉提示统一接口，而不只是文本接口。

### 3. [RefineAnything](https://arxiv.org/abs/2604.06870)

- 日期：2026-04-08
- 这篇在做什么：它把局部精修单独定义成任务，目标是只修指定区域，同时严格保持非编辑区域不变。
- 关键方法：论文提出 Focus-and-Refine，把目标区域 crop-resize 后集中分配分辨率预算，再用 blended-mask paste-back 和 boundary consistency loss 保背景稳定。
- 为什么重要：这篇很适合你们看，因为它直击 logo、文字、细线结构这些最容易塌掉的局部细节问题。
- 对系统设计的启发：对恢复系统来说，这基本就是 region repair loop 的论文版实现，尤其适合接到 verifier 后面做局部修复。

### 4. [TwiG](https://arxiv.org/abs/2511.16671)

- 日期：2025-11-20
- 这篇在做什么：这篇就是你说的“边画边想”代表作，它不是生成前想完，也不是生成后再修，而是在生成过程中持续插入推理。
- 关键方法：作者让文本推理与视觉生成交错进行，在生成局部区域前给出后续指导，同时对已经合成的区域进行反思。
- 为什么重要：这条路对 agentic editing 很关键，因为它意味着 planning、generation、reflection 可以在同一条链里滚动发生。
- 对系统设计的启发：如果你们要超过 Nano Banana，最值得学的不是单次提示增强，而是把 reasoning 嵌进生成过程本身。

### 5. [StruVis](https://arxiv.org/abs/2603.06032)

- 日期：2026-03-06
- 这篇在做什么：这篇走的是另一条“边想边画”路线：不直接生成中间图，而是先生成结构化视觉表示来帮助推理。
- 关键方法：它用 structured vision 作为中间 reasoning state，让 MLLM 在纯文本推理里也能感知视觉结构。
- 为什么重要：这代表另一种更省算力的路线，不一定非要真的出中间图，也可以先出结构化视觉草图。
- 对系统设计的启发：如果你们担心多轮生成太贵，可以把显式中间图换成结构化中间状态，先做 cheap planning 再做 expensive rendering。

### 6. [Nano Banana 2 Eval](https://arxiv.org/abs/2604.03061)

- 日期：2026-04-03
- 这篇在做什么：这篇论文正面回答通用图像编辑模型能不能替代传统 restoration，它直接拿 Nano Banana 2 做系统评测。
- 关键方法：论文强调 concise prompt 加 fidelity constraints 的重要性，并在多类退化、多种场景下和传统 restoration 方法对比。
- 为什么重要：结论很直接，在 full-reference 指标上表现非常强，感知质量也有竞争力，但对 prompt 很敏感，而且常常需要迭代 refinement。
- 对系统设计的启发：这基本证明了“编辑大模型 + 指令 + 迭代控制”能吃到 restoration 的一大块空间，但也说明 verifier、prompt parser 和 refinement loop 不能省。

## 今天最值得带走的结论

1. `CoVFT` 说明视觉侧 fine-tuning 依然有巨大空间，而且小模型未必天然输给更大模型。
2. `FlowInOne` 说明未来的编辑系统可能走向统一视觉接口，而不只是文本指令接口。
3. `RefineAnything` 说明 region-specific refinement 会成为 verifier 之后的标准后处理模块。
4. `TwiG / StruVis` 说明 reasoning 可以嵌入生成过程内部，形成真正的边画边想。
5. `Nano Banana 2 Eval` 说明通用编辑模型确实开始替代传统 restoration，但它替代的是单模型部分，不是整个系统工程.
